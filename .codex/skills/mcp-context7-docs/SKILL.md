---

## 目的

- ライブラリ仕様・推奨パターン・破壊的変更などを、推測ではなく一次情報に寄せて確認する。
- ただし API key 等の秘密情報は絶対に出力しない。

## いつ使うか

- 新規ライブラリ/未経験領域の実装をする前。
- エラーの原因がドキュメントの前提違いっぽいとき。
- “最近変わった可能性がある” 挙動を確認したいとき。

## 手順（概略）

1. `resolve-library-id` で対象ライブラリを特定する
2. `query-docs` で知りたい観点を検索する
3. 得られた情報を「今回のコードにどう反映するか」を短くまとめ、必要な変更だけ入れる

## 運用ルール

- API key やトークンがログに混ざる可能性があるため、出力は最小限にする。
- ドキュメント引用は短く（要点の要約を中心に）する。
- 実装反映時は “無関係差分を増やさない”。

## 出力（期待）

- 「何を調べたか」
- 「結論（今回の実装に必要なポイント）」
- 「それに基づく変更方針」
